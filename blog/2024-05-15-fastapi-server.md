## 背景

Microsoft Learn のクイックスタートは便利だが、サービスごとにページを見に行く必要があり面倒。コードベースを作っておいて、必要に応じて切り出して使えるネタ帳を作りたい。

Python を使うことが多いが、ランタイム環境やライブラリの依存関係を管理した状態で、Azure AI Services の API を叩く環境を作りたい。

## 方針

**フロントエンドとバックエンドを分離する**

シングルバイナリで実行できる形だと開発速度が最も速いと考えているが、技術選定の柔軟さや、依存関係を分離する目的でフロントエンドとバックエンドを分離する。
クラウドサービスの認証情報管理などはバックエンドに集約させて共通基盤として使うことで、フロントエンドの実装が簡略化できることもメリット。
バックエンド・フロントエンド双方実装することで、利用者・提供者双方の観点で API 設計の改善ループを回すことができる。

**コードからドキュメントを生成する**

基本原則として、コードがマスタとなりドキュメント・仕様書類が自動生成されるようなパイプラインを構築する。これにより、コードと仕様書の乖離を防ぐことができ、ワークフローが明確化される。

**API 仕様を OpenAPI で管理する**

OpenAPI スキーマは、実装と仕様書の乖離が起きないように、フレームワークから自動生成されることが望ましい。フレームワークの中には、OpenAPI スキーマからコードを生成するアプローチと、コードからスキーマを生成するアプローチがある。ここでは後者のアプローチを採用する。

**API クライアントライブラリを OpenAPI スキーマから自動生成する**

フロントエンドがバックエンド API を利用するためのクライアントライブラリを自動生成する。OpenAPI スキーマから API クライアントライブラリを自動生成することで、API 仕様の変更に追従しやすくなる。また、API クライアントライブラリを使うことで、API 仕様に準拠したリクエストを簡単に作成でき生産性が向上する。

**プログラミング言語**

基本的には得意なものを使うのがよいと考えているが、エコシステムの発展度合いは選定理由になりうると考える。
例えば、機械学習・画像処理・数値解析周りの拡張を使う場合や、クロスプラットフォームでのバイナリ生成が必要な場合などがある。
細かい差異はあれど、他の言語が適している場合は RPC 経由であったり、言語間のブリッジングを使って連携もできるので、言語の選定は柔軟に考える。
フロントエンドとバックエンドのプログラミング言語を統一しておくと、開発者のスキルセットが共通化されるため、開発効率が向上する。ビルドシステム周りも共通化できるため、運用コストが低減される。

**リポジトリ管理**

フロントエンドとバックエンドを分離するため、API 仕様の変更は双方で atomic に取り込む必要がある。規模が小さく、特にアクセスポリシーの管理などの事情が無い場合は、mono repo で管理する。規模が拡大したり、秘匿事項があり情報隠蔽などが必要な場合は、poly repo にして、Git submodule で分離する方法が考えられる。
(poly repo 向けに repo を分離する場合、変更取り込みのための submodule の更新作業などが発生するなど、運用コストが増える点は理解の上で選定する)

### 選定結果

- FastAPI: 実装から OpenAPI スキーマが生成される。言語は Python
- Streamlit: フロントエンド向けに採用。言語は Python
- mono repo: フロントエンドとバックエンドを同一リポジトリで管理

## 実装作業

### 開発環境の構築

- タスクランナーやスクリプトから環境構築ができるようにする
- Devcontainer を使って、開発環境をコンテナ化する
- VCS (GitHub 等)の設定ファイルを作成する
- Visual Studio Code の設定ファイルを作成する

これらは template repository として作成しておくと、プロジェクト立ち上げ時に手間を省くことができる。

### Git の設定

- .gitignore を設定する
- pre-commit を設定する

### GitHub の設定

- CODEOWNER を設定して、レビュープロセスを明示する
- Pull Request のテンプレートを設定する
- Issue のテンプレートを設定する
- ブランチの保護設定をする
- GitHub Actions を設定する
- LICENSE を明示する
- README を作成する
- Security Policy を設定する
- CONTRIBUTING を作成する

### CI/CD

- Dockerfile コンテナイメージを利用できるようにする。環境の差異を吸収し簡単に利用できるようにする。
- CI のタスクを自動化する。例えば、コードフォーマットのチェック、静的解析、テストの実行、コードカバレッジの取得など。
- CD のタスクを自動化する。例えば、tag を push すると GitHub Actions 経由でリリースが自動で行われるようにするなど。
- 各種タスクは Makefile のようなタスクランナーから簡単に実行できるようにする。不具合発生時に手間がかかると開発速度が低下するため。
- インフラ構築は理想順に IaC > スクリプト > 手順書のうちどれかで管理する。
- Bicep, AWS CDK, Terraform などの IaC ツールを使ってコード化すると再現性が高まり、手動での設定ミスを防ぐことができる。what-if / plan などの機能を使って、変更の影響範囲を事前に把握できる。

### バックエンドの実装

- FastAPI で API を実装する

### バックエンドコードからの自動生成パイプライン

- FastAPI から OpenAPI スキーマを自動生成する。FastAPI のフレームワークがサポートしている。
- OpenAPI スキーマから API クライアントライブラリを自動生成する。openapi-generator や Kiota などのツールを使う。

### フロントエンドの実装

- Streamlit でフロントエンドを実装する
